# Taste 产品需求文档 PRD v0.2（原生小程序 + Soft Morning UI 版）

> 本文档为 Taste 项目当前 PRD 正式版。  
> 当前阶段统一采用微信原生小程序开发，不采用 Taro / React / Vue 跨端方案。  
> 本版本在原生 MVP 基线之上，补充完整产品定位、UI 风格规范、页面视觉规则与组件设计约束，作为后续产品设计、开发实现与 AI 协作生成代码的统一依据。

---

# 0. 文档定位与基线约束

## 0.1 当前唯一技术基线

- 开发形态：微信原生小程序
- 页面技术：WXML + WXSS + JavaScript + 页面 JSON
- 数据存储：微信本地存储（`wx.setStorageSync / wx.getStorageSync`）
- 工程根目录：`miniprogram/`

## 0.2 一致性约束（必须遵守）

- 本文档不使用 `src/`、`app.tsx`、`React`、`Taro` 等跨端工程结构描述。
- 页面路由、目录结构、字段定义、技术栈说明必须保持同一工程形态。
- 若未来迁移框架，需新增版本文档（如 v0.3+）并明确迁移窗口，不在当前版本混用双方案。
- 所有视觉规范与 UI 组件落地必须优先适配微信原生小程序开发方式。
- UI 实现以 **ColorUI + Iconfont** 为基础样式体系，允许在其基础上做轻量二次封装，但不允许引入重型 UI 框架破坏项目的轻量性与一致性。

# 1. 产品概述

## 1.1 产品名称

- 中文名：口味
- 英文名：Taste

## 1.2 产品一句话描述

**以食物为入口，记录生活当下情绪与温度的小程序。**

## 1.3 产品核心理念

“口味”不仅代表食物真实的味道，也代表生活的味道、心情的味道、记忆的味道。  
用户记录的表面对象是“吃了什么”，真正被保留下来的是“那一刻的生活状态”。

产品希望传达的不是效率工具感，而是一种轻柔、温暖、让人愿意停留片刻的生活感受：

**一口食物，一口生活。**

## 1.4 产品愿景

打造一个轻量、私密、有温度的食物生活记录小程序。  
它不强调公开展示，不追求复杂社交，而是帮助用户用最轻松的方式记录普通日子里的情绪重量，并在未来回看时重新感受到那段生活。

# 2. 目标用户

## 2.1 核心用户画像

### 用户类型 A：喜欢记录生活的人
- 有记录日常、拍照、写短句的习惯
- 喜欢保留情绪、画面与日常片段
- 对食物和生活方式有感受力

### 用户类型 B：重视情绪价值的年轻用户
- 更关注产品是否“舒服”“有温度”“治愈”
- 不喜欢冷冰冰的工具型记录产品
- 希望记录本身就是一种轻松的体验

### 用户类型 C：希望回顾日常的人
- 会在未来回看“过去吃了什么、那时心情如何”
- 对普通生活中的细节记忆有需求
- 希望自己的记录有一定整理性与可回顾性

## 2.2 用户共同特征

- 重视生活感受，而非单纯数据记录
- 愿意接受轻量输入，不愿填写复杂表单
- 偏好低压力、柔和、直观的产品体验
- 对 UI 的气质与整体氛围较敏感

# 3. 产品定位

## 3.1 产品类型

轻量化、私密型、生活感导向的食物记录小程序

## 3.2 产品关键词

- 温暖
- 轻柔
- 私密
- 生活化
- 回看
- 情绪价值
- 食物入口
- 日常记录

## 3.3 与普通记录类产品的区别

Taste 的差异不在于功能更多，而在于记录视角与情绪价值。

### 3.3.1 Food as Entry, Life as Core
以食物为入口，核心记录的是生活状态。

### 3.3.2 Private, Not Performative
强调轻私密，不鼓励社交表演。

### 3.3.3 Emotional Weight in Ordinary Moments
重视普通时刻背后的情绪重量。

### 3.3.4 Looking Back Matters
回看能力是核心，而非附属。

# 4. 设计目标

## 4.1 产品目标

在 MVP 阶段，用最小可运行闭环验证以下事情：

- 用户是否能轻松创建一条记录
- 用户是否能自然浏览和回看记录
- 用户是否能在日历中感知自己的记录分布
- 产品是否能通过视觉和交互传达“温暖、生活化、有呼吸感”的独特气质

## 4.2 体验目标

Taste 的体验目标不是“高效录入”，而是“轻轻记下今天”。

用户打开小程序后应感受到：

- 界面柔和，不压迫
- 色彩温暖，不冰冷
- 操作轻量，不复杂
- 页面留白充足，有呼吸感
- 记录像写下一小段生活，而不是填写后台表单

# 7. UI 视觉设计总则

## 7.1 最终 UI 方向

本项目最终采用：

# **C. 晨光轻食 Soft Morning**

这是一套以“清晨自然光、低饱和暖色、轻柔简约、生活温度”为核心的视觉语言。

## 7.2 UI 关键词

- 晨光
- 温柔
- 低饱和暖色
- 轻呼吸感
- 柔和圆角
- 生活感
- 日常陪伴
- 轻私密
- 不科技
- 不冰冷

## 7.3 视觉气质定义

整体视觉应像清晨照进厨房和餐桌的自然光。  
不是厚重复古，也不是冰冷极简，而是一种**干净、温暖、轻柔、自然、耐看**的状态。

它应该给人的第一感觉是：

- 舒服
- 安静
- 柔和
- 没有攻击性
- 愿意停留
- 愿意记录

## 7.4 明确禁止的视觉方向

以下风格不允许在 Taste 中出现：

- 科技蓝、赛博风、霓虹风
- 过强对比的纯黑白极简
- 冷灰调办公后台感
- 过重拟物、过深阴影、厚重 3D 按钮
- 复杂炫技动效
- 高饱和荧光色大面积使用
- 过度商务化、企业系统化的视觉语言

# 8. UI 样式库方案

## 8.1 UI 实现基础

本项目 UI 样式体系采用：

- **ColorUI**
- **Iconfont**

共同组成 Taste 的小程序端 UI 样式库基础。

## 8.2 选择 ColorUI 的原因

ColorUI 适合作为微信小程序的轻量视觉基础层，原因包括：

- 适配原生小程序开发方式
- 提供基础布局、按钮、卡片、表单、标签等样式能力
- 能帮助快速建立一致的页面节奏
- 方便在 MVP 阶段提升页面完成度
- 二次定制成本相对较低

## 8.3 选择 Iconfont 的原因

Iconfont 用于建立项目统一图标体系，原因包括：

- 便于维护统一图标风格
- 适合页面中 tab、操作按钮、状态提示、记录类型等视觉表达
- 在小程序中接入灵活
- 易于后期扩展图标资源

## 8.4 使用原则

- ColorUI 作为**基础样式骨架**
- Iconfont 作为**统一图标来源**
- Taste 自定义视觉层作为**二次风格封装**
- 不允许直接堆砌 ColorUI 默认视觉效果而不做调性调整
- 所有 ColorUI 组件必须根据 Soft Morning 主题进行颜色、圆角、留白、阴影的二次规范
- Iconfont 图标必须统一选择圆润、简洁、生活化风格，避免锐利科技感图标

## 8.5 落地方式

建议形成以下样式分层：

### 基础层
- 微信原生组件能力
- ColorUI 通用布局类
- Iconfont 图标字体或图标资源

### 主题层
- `theme-soft-morning.wxss`
- 颜色变量
- 圆角变量
- 阴影变量
- 页面背景与卡片风格变量

### 业务组件层
- `record-card`
- `mood-tag`
- `empty-state`
- `calendar-grid`
- `soft-button`
- `soft-section-title`

# 9. 色彩系统（Soft Morning）

## 9.1 主色板

### 主背景色
- 牛奶白：`#FAF6F0`

### 次背景色
- 浅麦色：`#EADBC8`

### 主品牌暖色
- 杏桃暖橙：`#D9A77A`

### 深主色
- 暖焦糖：`#C58D61`

### 正文字色
- 深咖棕：`#4C4038`

### 次文字色
- 暖灰棕：`#8C786A`

## 9.2 辅助色

### 鼠尾草绿
- `#B7C4AE`

### 柔桃粉
- `#E7C7BC`

### 淡蜂蜜黄
- `#E7C978`

### 柔米卡片底色
- `#FFFDF9`

### 轻浅分层底色
- `#F5EDE3`

### 浅分隔底色
- `#EEE3D6`

## 9.3 功能色原则

Taste 不采用强烈工业型功能色体系，而采用柔化后的功能色表达：

### 成功 / 保存成功
- 暖绿色或鼠尾草绿系，避免荧光绿

### 提示 / 轻提醒
- 淡蜂蜜黄系，避免刺眼亮黄

### 警告 / 表单缺失
- 柔橙色系，避免鲜红色大面积使用

### 错误 / 删除风险
- 仅在必要场景使用偏柔和的红棕色，不允许纯高饱和大红作为主视觉

## 9.4 用色比例建议

- 70%：大面积浅背景（牛奶白 / 浅米色）
- 20%：卡片、分层、弱色块
- 10%：强调色（杏桃暖橙、柔桃粉、鼠尾草绿）

# 10. 字体与排版规范

## 10.1 字体气质

字体应呈现：

- 清晰
- 柔和
- 现代
- 不过于几何
- 不强硬

## 10.2 字号建议

### 页面主标题
- 24px - 28px
- 字重偏中高
- 用于首页欢迎文案、详情标题

### 模块标题
- 16px - 18px
- 用于分区标题

### 正文
- 14px - 16px
- 保证阅读舒适性

### 辅助说明
- 12px - 13px
- 用于日期、标签、状态说明

## 10.3 行高建议

- 标题：1.3 - 1.4
- 正文：1.6 - 1.8
- 说明文字：1.5 - 1.6

## 10.4 排版原则

- 保持留白，不堆信息
- 优先单列布局，减少压迫
- 文案节奏要轻
- 避免页面首屏塞满操作入口
- 每个模块之间需有明确呼吸空间

# 11. 圆角、阴影与材质规范

## 11.1 圆角规范

Taste 整体使用**偏大圆角**。

### 建议值
- 按钮：20px - 999px（胶囊）
- 卡片：18px - 26px
- 图片：16px - 24px
- 标签：999px
- 输入框：16px - 20px

## 11.2 阴影规范

阴影必须轻、软、浅，不允许厚重压迫感。

### 阴影原则
- 像晨光下的轻微浮起
- 强调柔和，不强调立体冲击
- 用于卡片、底部导航选中项、主按钮

### 阴影示例方向
- `0 8px 20px rgba(140,120,106,0.08)`
- `0 10px 25px rgba(140,120,106,0.10)`

## 11.3 材质原则

- 以“柔和纸面感、浅色卡片感”为主
- 不做玻璃科技风
- 不做金属质感
- 不做过强纹理
- 背景可以有轻微层次，但不应复杂

# 12. 图标系统规范（Iconfont）

## 12.1 图标风格要求

Iconfont 图标必须满足以下特征：

- 线条简洁
- 圆角感强
- 不尖锐
- 不未来感
- 不工业后台风
- 与生活记录产品气质一致

## 12.2 图标应用场景

- TabBar 图标
- 首页入口图标
- 创建页上传图标
- 回顾页筛选或搜索图标
- 详情页收藏 / 返回 / 分享图标
- 空状态插图辅助图标

## 12.3 图标选择规则

优先选择以下感觉的图标：

- 首页：房子 / 小屋 / 圆润 home
- 创建：加号 / 记录 / 编辑
- 回顾：日历 / 卡片 / 时间线
- 我的：用户 / 星星 / 小册子
- 返回：圆润箭头
- 分享：简单柔和分享图标

避免以下图标：

- 棱角过强
- 科技芯片感
- 办公软件感
- 商务后台线性图标套系
- 高复杂度多层图标

# 13. 组件视觉规范

## 13.1 Button 按钮

### 主按钮
- 颜色：暖焦糖 / 杏桃暖橙
- 文本：白色
- 形态：胶囊型
- 适用：保存、去记录、分享

### 次按钮
- 背景：卡片白 / 浅米色
- 文本：深咖棕
- 适用：返回、编辑、辅助操作

### 按钮体验要求
- 不做锐利直角
- 不做高饱和炫彩
- 点击反馈柔和
- 尺寸偏舒适，不拥挤

## 13.2 Card 卡片

卡片是 Taste 的核心视觉单元。

### 卡片规则
- 浅色背景
- 大圆角
- 轻阴影
- 图片与文本之间有舒适间距
- 内容不应过密

### 卡片应用
- 最近记录卡片
- 回顾页记录卡片
- 详情页信息块
- 空状态提示块

## 13.3 Input 输入框

Create 页中的输入框不应呈现“后台录入感”。

### 视觉要求
- 浅底色
- 柔和边界
- 足够内边距
- 焦点状态可带轻微暖色强调
- 占位文案语气温和

## 13.4 Tag 标签

### 用途
- 心情标签
- 时间标签
- 类型标签
- 状态标签

### 视觉要求
- 胶囊形
- 低饱和浅底色
- 文本颜色柔和
- 不做高对比刺眼色块

## 13.5 Empty State 空状态

空状态不是冷冰冰的“暂无数据”。

### 表达原则
- 语气温柔
- 可以带一句轻生活化文案
- 使用柔和插图 / 图标 / 浅色块辅助
- 引导用户去创建第一条记录

# 14. 页面级 UI 详细规范

## 14.1 Home 页面

### 页面目标
展示产品氛围与最近记录，作为进入记录与回看的入口。

### 页面气质
首页不是功能菜单页，而是“进入今天生活的一扇柔和入口”。

### 页面结构

#### 1）顶部区域
- 产品名：口味 Taste
- 可包含轻问候语
- 使用柔和留白，不拥挤

#### 2）主视觉文案区
- 核心文案：一口食物，一口生活。
- 辅助文案强调记录生活与情绪
- 可有主按钮“记录今天”

#### 3）轻入口区
- 早餐 / 午餐 / 晚餐等轻分类入口可选
- 以小卡片或柔和圆角块呈现

#### 4）最近记录区
- 显示最近 3-5 条记录
- 卡片式展示
- 点击进入详情页

#### 5）底部导航
- 首页 / 创建 / 回顾 / 我的

### MVP Requirements
- 可读取最近记录
- 无数据时显示空状态
- 存在“去记录”按钮

### Home 页面视觉重点
- 首屏必须舒服、轻、柔
- 不允许顶部大面积深色压迫
- 不做复杂 Banner 轮播
- 主视觉文字要有呼吸感

---

## 14.2 Create 页面

### 页面目标
让用户快速完成一条记录的创建。

### 页面气质
Create 页应像“邀请用户轻轻写下今天的味道”，而不是“让用户填写字段”。

### 字段

- 图片（MVP 支持 1 张）
- 标题
- 记录类型
- 心情
- 描述
- 日期（默认今天）

### Record Type Options

- `cook`：自己做的
- `eat-out`：外面吃的
- `takeout`：外卖
- `moment`：生活瞬间

### Mood Options

- `happy`：开心
- `calm`：平静
- `sad`：难过
- `tired`：疲惫
- `anxious`：焦虑

### Create 页视觉要求

#### 1）表单层级轻
- 使用卡片式分组
- 每一块之间保留留白
- 输入项要简洁直观

#### 2）图片上传区
- 使用浅米色 / 虚线边框 / 圆角
- 不要做过于技术化的上传框

#### 3）心情选择区
- 使用柔和标签 / 胶囊式按钮
- 可选中后变为暖色高亮

#### 4）主操作区
- 页面底部或末尾有明确“保存这份味道”按钮
- 主按钮需要有温暖感和完成感

### MVP Requirements
- 表单完整可输入
- 支持选择图片
- 支持保存到本地
- 保存成功后跳转到回顾页

## 14.3 Review 页面

### 页面目标
展示全部记录历史，形成最基础回看能力。

### 页面气质
Review 页不是冷冰冰的“历史列表”，更像“生活回顾”。

### 页面结构

#### 1）顶部标题区
- 页面标题：回顾 / 我的记录
- 可带月份或时间提示

#### 2）筛选辅助区（MVP 可轻量）
- 可预留全部 / 平静 / 开心等标签筛选位
- 即使 MVP 不做完整筛选，UI 结构可预留

#### 3）记录列表区
- 倒序展示记录
- 卡片式布局
- 显示图片、标题、心情、日期

### Review 页视觉要求
- 列表节奏不能过于拥挤
- 图片缩略图要圆角统一
- 每张卡片的信息密度适中
- 需要有“回看过去”的温柔感

### MVP Requirements
- 支持空状态
- 支持卡片列表
- 显示图片、标题、心情、日期

---

## 14.4 Calendar 页面

### 页面目标
通过月历可视化用户记录密度。

### 页面气质
Calendar 页应像轻柔归档工具，而不是复杂日程管理系统。

### 页面结构

#### 1）月份头部
- 当前年月
- 左右切换月按钮（后续可扩展）

#### 2）日期网格
- 当前月的日历网格
- 有记录日期加柔和标记

#### 3）日期点击反馈
- MVP 可 toast 或 console
- 后续可进入当天记录页

### Calendar 页视觉要求
- 日期格子保持干净
- 标记方式轻柔，不可强烈抢眼
- 当前日期、有记录日期、普通日期要有明确但克制的区分
- 不做复杂大面积强色背景

### MVP Requirements
- 正确显示当前月网格
- 可根据记录判断某天是否有内容

## 14.5 Detail 页面

### 页面目标
完整展示某条记录。

### 页面气质
Detail 页应像“阅读一段生活片段”，而不是查看数据明细。

### 页面结构

#### 1）顶部操作区
- 返回
- 收藏 / 分享（可预留）

#### 2）主图区域
- 大图展示食物或当时画面
- 圆角大图

#### 3）标题与基础信息
- 标题
- 日期
- 类型
- 心情标签

#### 4）描述正文
- 展示用户记录内容
- 行高舒适，适合阅读

#### 5）补充备注区
- 可选小备注
- 后续可扩展编辑 / 分享功能

### Detail 页视觉要求
- 主图要有存在感
- 文本区像阅读区，而不是说明书
- 标签、时间、地点信息应整齐但轻量
- 页面整体要有“保留回忆”的感觉

### MVP Requirements
- 根据 id 查询记录
- 找不到记录时给出兜底提示

作为项目说明与后续设置入口。

轻说明页，不做复杂账户体系。

### 页面内容
- 产品名
- 版本号
- 项目说明
- Coming Soon 功能预告

### MVP Requirements
